## 8. Anti-patrones a evitar

### 8.1. Anemic Domain Model
Entidades que son solo datos, sin behavior. La validaciأ³n, transiciones de estado, cأ،lculos viven en services. **MAL.** Esa lأ³gica va en la entidad.

### 8.2. Interface Astronaut
Crear una interface para cada clase "por si acaso". Las interfaces existen para:
- (a) tener mأ؛ltiples adapters (`PrismaXRepo` + `InMemoryXRepo` para tests), o
- (b) invertir la dependencia entre capas.
Si no cumple ni (a) ni (b), **NO crear interface**.

### 8.3. Leaky Abstraction
Definir un port que retorna `Prisma.XxxGetPayload<...>`. Eso no es un port â€” es un wrapper de Prisma.

### 8.4. Service haciendo lأ³gica de negocio
Si en `mortality.service.ts` ves un `if (input.count > ...) throw ...`, esa lأ³gica **deberأ­a estar en `Mortality.log()`** (la entidad). El service solo orquesta.

### 8.5. Premature abstraction
Crear `IEmailService` con un solo `SendgridEmailService` "por si cambiamos de proveedor". Eso es deuda preventiva. Cuando aparezca el segundo proveedor, reciأ©n ahأ­ extraأ©s la interface.

### 8.6. Filtrar input â‰  lأ³gica de negocio

El principio "use cases orquestan, no calculan" (R7, آ§8.4) NO significa "use cases no condicionan". Filtrar QUأ‰ entities procesar es **orquestaciأ³n legأ­tima**, no lأ³gica de negocio.

`if (target.status === 'VOIDED') continue` en un use case es **orquestaciأ³n legأ­tima** â€” decidir QUأ‰ entities procesar es parte del rol del use case. Lo prohibido es **CأپLCULO de negocio** (aritmأ©tica sobre estado, transiciأ³n de estado, derivaciأ³n de campos).

| Permitido en use case (orquestaciأ³n) | Prohibido en use case (cأ،lculo â€” va al entity) |
|--------------------------------------|------------------------------------------------|
| `if (!target) continue`              | `newPaid = currentPaid + amount`               |
| `if (target.status === 'VOIDED') continue` | `newStatus = newPaid === total ? 'PAID' : 'PARTIAL'` |
| `if (input.dryRun) return preview`   | `if (newPaid > total) throw`                   |
| Loop con skip condicional            | `total = items.reduce(...)`                    |
| Componer load â†’ mutate â†’ persist     | Cualquier cambio en propiedades del entity     |

**Test mental**: si el cأ³digo del use case toca propiedades calculadas del entity (`paid`, `balance`, `status`, `total`) para SETEARLAS, es cأ،lculo y debe ir al entity. Si solo las LEE para decidir si invoca o no al entity, es orquestaciأ³n.
