## 1. Architectural shape

The saga is a per-request, 3-step state machine. There is NO shared state, NO worker, NO outbox — it lives entirely inside the Next.js API request lifecycle (Node.js runtime). Compensation is synchronous and adjacent to the try/catch.

### Canonical state machine (applies to all three operations)

```
[Validate/Guard] -> [DB primary write] -> [Clerk call] -- ok --> [COMMITTED] (200/201)
                                              |
                                              v fail (non-idempotent)
                                     [Compensate (DB)] -- ok --> [COMPENSATED] (503 ExternalSyncError)
                                              |
                                              v fail
                                         [DIVERGENT] (503 ExternalSyncError + divergent log)
```

Clerk idempotency short-circuit applies on the Clerk edge:
- `createOrganizationMembership` error classified as `already_a_member_in_organization` -> COMMITTED.
- `deleteOrganizationMembership` error classified as `resource_not_found` (404) -> COMMITTED.

### Per-operation state table

| Operation | DB primary write | Clerk call | Compensation on Clerk fail | Commit log | Compensate log | Divergent payload |
|---|---|---|---|---|---|---|
| add-new | `repo.addMember({ organizationId, userId, role }, tx?)` | `createOrganizationMembership({ organizationId: clerkOrgId, userId: clerkUserId, role: "org:member" })` | `repo.hardDelete(organizationId, memberId)` | `members.clerk_sync.committed` `{op:"add"}` | `members.clerk_sync.compensated` `{op:"add"}` | `{dbState:"member_inserted", clerkState:"membership_absent"}` |
| add-reactivate | `repo.reactivateMember(organizationId, memberId, role)` (was deactivated -> active) | same `createOrganizationMembership` | `repo.deactivateMember(organizationId, memberId)` (re-set `deactivatedAt`) | same, `{op:"reactivate"}` | same, `{op:"reactivate"}` | `{dbState:"member_active", clerkState:"membership_absent"}` |
| remove | `repo.deactivateMember(organizationId, memberId)` | `deleteOrganizationMembership({ organizationId: clerkOrgId, userId: clerkUserId })` | `repo.reactivateMember(organizationId, memberId, previousRole)` | same, `{op:"remove"}` | same, `{op:"remove"}` | `{dbState:"member_deactivated", clerkState:"membership_present"}` |

For `remove`, `previousRole` MUST be captured from `member.role` BEFORE deactivation (pre-flight `findMemberById` already returns it).

## 2. Saga helper shape

### Options considered

1. **Inline try/catch blocks** (3 explicit saga bodies in `members.service.ts`). Readable at the call site, no abstraction cost, but the error-classification / logging ceremony repeats 3x, which invites drift (one of the original bugs was exactly the asymmetric swallow between add-new and reactivation — repetition is how that slipped in).
2. **Generic `runSaga<T>({ dbWrite, externalCall, compensate, onDivergence })`** in a new `features/organizations/member-clerk-saga.ts`. Keeps the 3 operation bodies lean and makes the divergence contract a single function.
3. **Class `MemberClerkSaga` encapsulating three named methods**. More scaffolding, no win over a helper function.

### Decision: Option 2 — a single helper function, scoped to members

**File**: `features/organizations/member-clerk-saga.ts` (NEW). NOT `features/shared/saga.ts` — we are explicitly rejecting premature generalization per proposal §3 ("this service is the template every future external-API integration will copy" — a template, not a framework). When `iva-books`/ANAF or any second consumer lands, the helper will already have had a real production user exercising every branch; generalizing from two known shapes is safer than from one hypothetical shape.

**Signature**:

```ts
// features/organizations/member-clerk-saga.ts
import "server-only";
import { isClerkAPIResponseError } from "@clerk/shared/error";
import { ExternalSyncError } from "@/features/shared/errors";

export type MemberSagaOperation = "add" | "reactivate" | "remove";

export type MemberSagaContext = {
  operation: MemberSagaOperation;
  organizationId: string;
  memberId: string;       // filled after primary write for add-new; known up-front otherwise
  clerkUserId: string;
  correlationId: string;  // crypto.randomUUID(), generated by caller
};

export type MemberSagaInput<T> = {
  ctx: MemberSagaContext;
  dbWrite: () => Promise<{ memberId: string; result: T }>;
  clerkCall: () => Promise<void>;
  compensate: () => Promise<void>;
  // Classifiers for idempotent Clerk success
  isIdempotentSuccess: (err: unknown) => boolean;
  // Divergent payload — caller-specific db/clerk state strings
  divergentState: { dbState: string; clerkState: string };
};

export async function runMemberClerkSaga<T>(input: MemberSagaInput<T>): Promise<T> { /* ... */ }
```

Helper contract (pseudocode):

```
1. { memberId, result } = await dbWrite()   // primary; on throw bubbles as 500 via AppError
2. ctx.memberId = memberId
3. try { await clerkCall() }
   catch (clerkErr) {
     if (isIdempotentSuccess(clerkErr)) { logCommitted(ctx); return result; }
     try { await compensate(); logCompensated(ctx, clerkErr); }
     catch (compErr) { logDivergent(ctx, clerkErr, compErr); throw new ExternalSyncError(...) }
     throw new ExternalSyncError(...)  // single-failure 503
   }
4. logCommitted(ctx); return result
```

**Rationale** (readability/test-surface/extensibility):
- **Readability**: the three methods in `MembersService` shrink to linear "gather inputs -> call `runMemberClerkSaga` -> return DTO" shape. Branching lives in one place.
- **Test surface**: one helper to unit-test for the four failure-matrix rows (a/b/c/d); the three service methods only exercise "I wired up the right callbacks". Mocks become a single injected object per scenario.
- **Extensibility**: the shape is sized to the KNOWN problem (two-system write with compensation). If `iva-books` ANAF lands with a different shape (e.g. pre-commit queue, different idempotency semantics), we extract a truly-generic helper THEN.
- **Risk-of-drift closed**: the asymmetric-swallow bug from the old code literally cannot recur because all three operations share one `isIdempotentSuccess` -> `compensate` -> `logDivergent` path.

### Package and version

- `@clerk/nextjs: ^7.0.7` (root dep); re-exports `clerkClient` from `@clerk/nextjs/server`.
- Transitive: `@clerk/backend: 3.2.3` (the actual SDK doing HTTP calls and throwing errors).

### Method signatures (verified from `node_modules/@clerk/backend/dist/api/endpoints/OrganizationApi.d.ts`)

```ts

  organizationId: string;
```

Both take the **Clerk org id** (our `organization.clerkOrgId`) and the **Clerk user id** (our `user.clerkUserId`) — both already threaded through today's code.

### Error shape — `ClerkAPIResponseError`

Verified from `node_modules/@clerk/shared/dist/runtime/index-DS9PyRNS.d.ts`:

```ts
class ClerkAPIResponseError extends ClerkError {
  status: number;                  // HTTP status — 404/422/etc.
  clerkTraceId?: string;           // propagate to logs
  retryAfter?: number;             // seconds — when Clerk 429s
  errors: ClerkAPIError[];         // array; first element is canonical
}
interface ClerkAPIError {
  code: string;                    // stable Clerk error code
  message: string;
  longMessage?: string;
  meta?: Record<string, unknown>;
}
// Guard:
isClerkAPIResponseError(error: unknown): error is ClerkAPIResponseError;
```

### Classifier helpers (replaces today's ad-hoc `isClerkDuplicateError`)

New file: `features/organizations/clerk-error-classifiers.ts`.

```ts
import { isClerkAPIResponseError } from "@clerk/shared/error";

const DUPLICATE_CODES = ["already_a_member_in_organization", "duplicate_record"];
const NOT_FOUND_CODES = ["resource_not_found", "organization_membership_not_found"];

export function isClerkDuplicateMembershipError(err: unknown): boolean {
  if (!isClerkAPIResponseError(err)) return false;
  const code = err.errors?.[0]?.code;
  return (
    DUPLICATE_CODES.includes(code ?? "") ||
    // Defensive: match substring as the existing helper did, to tolerate minor Clerk drift
    (code?.includes("duplicate") ?? false) ||
    (code?.includes("already_a_member") ?? false)
  );
}

export function isClerkMembershipNotFoundError(err: unknown): boolean {
  if (!isClerkAPIResponseError(err)) return false;
  if (err.status === 404) return true;
  const code = err.errors?.[0]?.code;
  return NOT_FOUND_CODES.includes(code ?? "");
}

export function clerkRetryAfterSeconds(err: unknown): number | undefined {
  if (!isClerkAPIResponseError(err)) return undefined;
  return err.retryAfter;
}

export function clerkErrorFingerprint(err: unknown):
  | { code?: string; status?: number; traceId?: string }
  | { code: "non_clerk_error"; message: string } {
  if (isClerkAPIResponseError(err)) {
    return { code: err.errors?.[0]?.code, status: err.status, traceId: err.clerkTraceId };
  }
  return { code: "non_clerk_error", message: err instanceof Error ? err.message : String(err) };
}
```

### Idempotency guarantees (documented assumption)

- `createOrganizationMembership` — Clerk returns 422 with code `already_a_member_in_organization` (or similar; see R-1 below) if the user already belongs to the org. We treat this as COMMITTED. On network timeout retry, either Clerk succeeds (membership created) or returns the duplicate code — both lead to COMMITTED.
- `deleteOrganizationMembership` — Clerk returns 404 (`resource_not_found`) if the membership is already absent. We treat this as COMMITTED. Retrying after network timeout is safe.

### Contract test (addresses R-1 from proposal)

Because we are assuming specific code strings and the classifier drifts silently if Clerk renames them, apply phase MUST add `features/organizations/__tests__/clerk-error-contract.test.ts`:

- Fixtures `__fixtures__/clerk-*.json` capturing real `ClerkAPIResponseError.toJSON()` shapes for: (a) duplicate membership add, (b) delete not-found, (c) 429 rate limit with `retryAfter`. Fixtures are CAPTURED ONCE against the real Clerk API in a dev env (documented procedure in the test file header), then committed.
- Tests assert `isClerkDuplicateMembershipError`, `isClerkMembershipNotFoundError`, `clerkRetryAfterSeconds` return the expected values for each fixture.
- If a fixture mismatches after a Clerk SDK upgrade, CI surfaces it as a failing contract test — design intent.

**Contract-test gap noted for apply phase**: fixtures (a/b/c) must be captured from a real Clerk API call once. Apply phase records this as a blocking step before merging.

## 4. `ExternalSyncError` type

Added to `features/shared/errors.ts` following the existing `AppError` subclass pattern.

```ts
export type DivergentState = {
  dbState: string;        // e.g. "member_deactivated"
  clerkState: string;     // e.g. "membership_present"
};

export type ExternalSyncErrorDetails = {
  divergentState: DivergentState;
  operation: "add" | "reactivate" | "remove";
  correlationId: string;
  retryAfterSeconds?: number;
  clerkErrorCode?: string;
  clerkTraceId?: string;
};

export class ExternalSyncError extends AppError {
  constructor(message: string, details: ExternalSyncErrorDetails) {
    super(message, 503, EXTERNAL_SYNC_ERROR, details);
  }
}

// Error-code registry (goes in the RBAC section or a new "External Sync" section):
export const EXTERNAL_SYNC_ERROR = "EXTERNAL_SYNC_ERROR";
```

### Error-code registry observation

Reviewed `features/shared/errors.ts`. The "7 intentionally unused codes" framing is incorrect — the existing codes are single-string exports consumed by their respective feature modules; there are no "reserved unused" slots to claim. `EXTERNAL_SYNC_ERROR` is added as a new top-level constant in a new `// External Sync` section, matching the existing pattern (see `LEGACY_CLOSE_REMOVED`, `CANNOT_CHANGE_OWN_ROLE`). Error-serialization spec REQ-5 (at archive time) gets `EXTERNAL_SYNC_ERROR` added to its code table, and REQ-1 gets `ExternalSyncError` appended to its subclass list.

### Serialization (verified by reading `features/shared/http-error-serializer.ts`)

`handleError` already serializes any `AppError` subclass via `statusCode` (lines 11–19). No route-level mapping. `ExternalSyncError` inherits this automatically. The `details` object (including `divergentState`, `correlationId`, `retryAfterSeconds`) is echoed in the response body via the existing `...(error.details ? { details: error.details } : {})` spread. **I-5 confirmed.**

Open question resolved: `retryAfterSeconds` travels in `details`, not as an HTTP `Retry-After` header. Reason: the existing serializer returns a plain JSON body without setting response headers beyond content-type. Adding a header path is out-of-scope for this change (would alter the serializer contract for all errors). Spec scenario S-MCS.4-2 asserts `details.retryAfterSeconds` only, consistent with this decision.

## 5. Observability

### Logger

**Decision**: keep `console.error` / `console.warn` / `console.info` with a structured JSON payload — no new logger dep.

Rationale: there is no `pino`/`winston`/`bunyan` in `package.json`. The codebase's existing pattern for structured-ish logging is `console.error("tag:", payload)` (see `features/organizations/members.service.ts:215`, `features/monthly-close/monthly-close.service.ts:139` uses `correlationId` directly). Introducing a logger is out of scope per proposal §4 non-goals. Structured JSON body is sufficient for grep/filter in the serverless logs.

### Log API (new file `features/organizations/member-clerk-saga.logger.ts`)

```ts
type SagaLogBase = {
  event: "members.clerk_sync.committed"
       | "members.clerk_sync.compensated"
       | "members.clerk_sync.divergent";
  operation: "add" | "reactivate" | "remove";
  organizationId: string;
  memberId: string;
  clerkUserId: string;
  correlationId: string;
};
type CommittedLog = SagaLogBase & { event: "members.clerk_sync.committed" };
type CompensatedLog = SagaLogBase & {
  event: "members.clerk_sync.compensated";
  clerkError: { code?: string; status?: number; traceId?: string };
};
type DivergentLog = SagaLogBase & {
  event: "members.clerk_sync.divergent";
  dbState: string;
  clerkState: string;
  clerkError: { code?: string; status?: number; traceId?: string };
  compensationError: { name: string; message: string };
};

export function logCommitted(l: CommittedLog): void {
  console.info(JSON.stringify(l));
}
export function logCompensated(l: CompensatedLog): void {
  console.warn(JSON.stringify(l));
}
export function logDivergent(l: DivergentLog): void {
  console.error(JSON.stringify(l));
}
```

- **Sampling**: none. Every divergence matters (operator-reconciliation signal).
- **PII**: `email` is deliberately NOT in the log payload (already referenced via `memberId`+`organizationId`).
- **Correlation**: `correlationId = crypto.randomUUID()` generated once per saga run in the service method, passed into the helper. Pattern reused from `features/monthly-close/monthly-close.service.ts:139`.

### Spec field-mapping (REQ-MCS.5 realization)

| Spec field | Log field | Source |
|---|---|---|
| `memberId` | `memberId` | DB primary write result (add) or pre-flight `findMemberById` (reactivate/remove) |
| `clerkUserId` | `clerkUserId` | `user.clerkUserId` |
| `organizationId` | `organizationId` | route param -> service arg |
| `operation` | `operation` | literal, set per method |
| `dbState` | `dbState` | per-operation table §1 |
| `clerkState` | `clerkState` | per-operation table §1 |
| `correlationId` | `correlationId` | `crypto.randomUUID()` |

## 8. Migrations / schema

**No schema changes.** Soft-delete is already represented via `OrganizationMember.deactivatedAt` (invariant I-1). `hardDelete(organizationId, memberId)` is a new repo method, not a schema change — it issues `DELETE FROM OrganizationMember WHERE id = ? AND organizationId = ?` via Prisma's `delete`/`deleteMany` API.

No `previousRole` column is added. `previousRole` is captured into request-local scope from `findMemberById` before deactivation and passed to the compensation `reactivateMember` call. This is sufficient because compensation is synchronous in the same request.

## 11. Spec-surfaced clarification — RESOLVED

**Question**: does `reactivateMember` accept a `role` param, or preserve the prior role? What does REQ-MCS.3 compensation need?

**Evidence** (read from `features/organizations/organizations.repository.ts:199-207`):

```ts
async reactivateMember(memberId: string, role: string): Promise<OrganizationMember> {
  return this.db.organizationMember.update({
    where: { id: memberId },
    data: { deactivatedAt: null, role },
  });
}
```

Signature: `(memberId, role)`. It DOES accept `role` and overwrites the stored role. It is also NOT organizationId-scoped (I-8 concern carried forward).

**Decision: Option (a) — capture `previousRole` pre-deactivation + extend signature.**

- Change signature to `reactivateMember(organizationId, memberId, role, tx?)`.
- `removeMember` service method reads `member.role` from the pre-flight `findMemberById` result into `const previousRole = member.role;` BEFORE calling `deactivateMember`. Compensation calls `repo.reactivateMember(organizationId, memberId, previousRole)`.
- `addMember` reactivation branch passes the **new** `role` arg (current behaviour) to `reactivateMember`, and on compensation the repo method called is `deactivateMember(organizationId, memberId)` — no role needs re-capture for this branch.
- Only caller today is `members.service.ts:105`; signature change is safe.

**Why not (b)** — no `previousRole` column exists; adding one is a schema change and is explicitly out-of-scope (§8).
**Why not (c)** — "undo within same request" is exactly what the saga IS; no additional flow needed.

## 12. Out-of-scope reaffirmation

Mirrored from proposal §2 and spec §Out-of-Scope — this design introduces NONE of the following:

- Clerk webhook ingestion / reconciliation.
- Transactional outbox / background worker / scheduler.
- Automatic retry of the Clerk call inside the request.
- Daily divergence-detection job.
- `updateRole` Clerk sync (verified: no Clerk calls).
- `createOrganization` / `syncOrganization` flows.
- New logger dependency (`pino`/`winston`) — `console.*` with JSON body is sufficient.
- `Retry-After` HTTP header — `details.retryAfterSeconds` in the JSON body only (serializer contract unchanged).
- Changes to the `handleError` serializer (the existing generic `AppError` branch suffices).
- `isSystem` role flag impact on add/remove.

End of design.
